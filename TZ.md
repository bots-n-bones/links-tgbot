# Техническое задание: Link Collector

**Версия:** 1.1  
**Дата:** 10.07.2026  
**Статус:** Draft

---

## 1. Общее описание

### 1.1. Проблема

Команда активно обменивается полезными материалами (ссылками) в Telegram-чатах. Сообщений много — нет времени просматривать, сортировать и возвращаться к накопленному. Ссылки теряются в потоке чата, отсутствует единый каталог с контекстом и поиском по темам.

### 1.2. Решение

Система из двух частей:

1. **Telegram-бот** — собирает ссылки из чатов и личных сообщений, обрабатывает их через LLM, отвечает пользователю о статусе.
2. **Web-дашборд** — HTML-интерфейс для просмотра, фильтрации, поиска, тематических подборок и вопросов к базе знаний.

В перспективе дашборд интегрируется в корпоративный сайт. На этапе MVP — автономное решение без авторизации.

### 1.3. Цели

- Автоматически собирать и каталогизировать ссылки из Telegram.
- Давать каждой ссылке описание и теги с помощью OpenAI.
- Показывать популярность материалов внутри команды.
- Позволять искать по базе на естественном языке («есть ли у нас что-то про RAG?»).
- Генерировать тематические подборки за период.
- По запросу собирать расширенный LLM-отчёт по теме с внешними источниками.

## 2. Ограничения и допущения

| Параметр | Решение |
|----------|---------|
| Количество чатов | До 10 |
| Способ чтения сообщений | Только официальный Telegram Bot API |
| История сообщений | Только с момента добавления бота в чат |

| Авторизация дашборда | Нет на MVP (интеграция в сайт — позже) |

| Хостинг | VPS, Docker Compose |
| Язык интерфейса бота | Русский |

### 2.1. Требования к Telegram-боту

- Бот добавляется администратором в каждый из целевых чатов вручную.
- В BotFather: **Privacy Mode отключён** (`/setprivacy` → Disable) — иначе бот не видит обычные сообщения в группах.
- Бот обрабатывает только пользователей из whitelist (`ALLOWED_USER_IDS` в конфигурации).

## 3. Источники данных

### 3.1. Групповые чаты (до 10)

Бот читает все сообщения в чатах, где он добавлен. Из каждого сообщения извлекаются URL.

### 3.2. Личные сообщения боту

Пользователь из whitelist может напрямую отправить боту:

- сообщение со ссылкой;
- ссылку с комментарием;
- пересланное сообщение со ссылкой;
- несколько ссылок в одном сообщении.

Если сообщение не содержит ссылки и выглядит как вопрос — маршрутизируется в Q&A (см. §6.4).

### 3.3. Извлечение ссылок

- URL из Telegram entities (`url`, `text_link`).
- URL из текста сообщения (regex).
- URL из caption (подпись к медиа).
- URL из пересланных сообщений.

## 4. Функциональные требования

### 4.1. Сбор и очередь

| ID | Требование |
|----|------------|
| F-01 | Бот сохраняет в очередь (`raw_messages`) только сообщения, содержащие хотя бы один URL (по критериям §3.3). Сообщения без ссылок в групповых чатах не сохраняются и не занимают очередь (Privacy Mode отключён, поэтому бот технически видит все сообщения — фильтрация по наличию URL происходит до записи в БД). |
| F-02 | Идемпотентность: пара `(chat_id, message_id)` уникальна — повторная обработка невозможна. |
| F-03 | Обработка в двух режимах: **live** (сообщение сразу в очередь) и **batch** (Celery Beat 2× в день для необработанного). |
| F-04 | Worker обрабатывает очередь: извлечение → дедупликация → fetch meta → LLM → сохранение. |

### 4.2. Дедупликация ссылок

| ID | Требование |
|----|------------|
| F-10 | Перед полной обработкой URL нормализуется (схема, lowercase host, удаление utm/fbclid, trailing slash). |
| F-11 | Уникальность по `url_hash = sha256(normalized_url)` на уровне БД. |
| F-12 | Если ссылка уже есть — **не** создавать новую запись, **не** повторять LLM-обработку. |
| F-13 | Добавлять новый `link_source` (контекст: чат, отправитель, текст сообщения). |
| F-14 | Инкрементировать `source_count` и пересчитывать `unique_senders`, `priority_score`. |

### 4.3. Обработка ссылок (LLM)

| ID | Требование |
|----|------------|
| F-20 | Fetch метаданных страницы: title, og:description, favicon, domain. |
| F-21 | При ошибке fetch (403, timeout, paywall) — fallback на контекст из сообщения Telegram, статус `fetch_failed`. |
| F-22 | OpenAI генерирует: `description` (1–2 предложения), `tags` (1–3 тега), `confidence` (0.0–1.0). |
| F-23 | Теги свободные, но проходят нормализацию (lowercase, латиница, синонимы). Соответствия синонимов (`ии` → `ai`, `дизайн` → `design` и т.д.) хранятся в таблице `tag_synonyms` (§7.1), а не хардкодятся в коде — это позволяет пополнять список через дашборд/миграцию без деплоя. |
| F-24 | После обработки — генерация embedding (OpenAI) для векторного поиска. |
| F-25 | Лимит текста страницы для LLM: первые 3000–5000 символов. |

### 4.4. Счётчики и приоритет

| ID | Требование |
|----|------------|
| F-30 | `source_count` — общее число добавлений ссылки. |
| F-31 | `unique_senders` — число уникальных отправителей. |
| F-32 | `priority_score` пересчитывается при каждом новом source. |
| F-33 | Бейдж **«Популярно»** при `unique_senders >= 2` или `source_count >= 3`. |

**Формула приоритета:**

```
priority_score = source_count × 1.0
               + unique_senders × 2.0
               + recency_bonus
```

`recency_bonus` — экспоненциальный распад по давности последнего добавления:

```
recency_bonus = 3.0 × exp(-days_since_last_source / 7)
```

где `days_since_last_source` — число дней с даты последнего `link_sources.created_at` для этой ссылки. Свежая ссылка (0 дней) даёт бонус ~3.0, через 7 дней — ~1.1, через 14 дней — ~0.4. Пересчитывается при каждом новом source и лениво — раз в сутки батчем (Celery Beat) для всех ссылок, чтобы бейдж «Популярно» и сортировка не «протухали» без новых добавлений.

### 4.5. Ответы бота

| ID | Требование |
|----|------------|
| F-40 | При новой ссылке: подтверждение «Добавил», теги (после обработки), «Первая в базе». |
| F-41 | При дубле: «Уже в базе», счётчик добавлений и уникальных отправителей, теги. |
| F-42 | При нескольких ссылках в сообщении — сводка по каждой. |
| F-43 | При отсутствии ссылки: «Не нашёл ссылку» + подсказка `/help`. |
| F-44 | Пользователь не из whitelist: «Нет доступа». |
| F-45 | После обработки в личке — уведомление с описанием, тегами, счётчиком, ссылкой на дашборд. |

**Команды бота:**

| Команда | Описание |
|---------|----------|

|----|------------|

| F-64 | UI: прогресс «ищу… пишу отчёт…» (30–60 сек). |
| F-65 | Возможность добавить найденные ссылки в основную базу. |

### 4.8. Тематические подборки

| ID | Требование |
|----|------------|
| F-70 | Автогенерация 1× в неделю (Celery Beat, понедельник 09:00 MSK — настраиваемо). |
| F-71 | Группировка ссылок по тегам за последние 7 дней. |
| F-72 | LLM формирует markdown-подборку: главное, топ по востребованности, «не пропустить». |
| F-73 | В подборку попадают ссылки с достаточным `priority_score` или топ-N по тегу. |
| F-74 | Раздел **«Подборки»** в дашборде. |
| F-75 | Отправка digest в Telegram (`/digest` или авто в личку — настраиваемо). |

### 4.9. Вопросы к базе (Q&A, RAG)

| ID | Требование |
|----|------------|
| F-80 | Пользователь задаёт вопрос на естественном языке: «есть ли у нас что-то про RAG?». |
| F-80 | Доступно в дашборде (поле «Спросить базу…») и в боте (`/ask` или текст в личке без URL). |
| F-81 | Pipeline: embedding вопроса → pgvector similarity search → топ 5–10 ссылок → GPT → ответ с цитатами. |
| F-82 | Ответ содержит только ссылки из собственной базы, без выдуманных URL. |
| F-83 | В ответе указываются счётчики популярности («добавляли 3 раза»). |
| F-84 | Логирование в `qa_logs` (вопрос, ответ, matched_link_ids). |

## 5. Нефункциональные требования

### 5.1. Производительность

- Обработка одной ссылки (fetch + LLM + embedding): до 30 сек.
- Q&A: ответ до 10 сек.
- Research report: до 60 сек.

### 5.2. Надёжность

| ID | Требование |
|----|------------|
| NF-01 | Статусы ссылки: `pending` → `fetching` → `processing` → `done` / `failed`. |
| NF-02 | Retry с exponential backoff (до 3 попыток) для fetch и OpenAI. |
| NF-03 | Healthcheck endpoint: `GET /health`. |
| NF-04 | Алерт администратору в Telegram при 10+ последовательных `failed`. |
| NF-05 | Соблюдение лимитов Telegram Bot API (~30 сообщений/сек глобально, 1 сообщение/сек на чат): очередь исходящих ответов бота с троттлингом, особенно при батч-обработке (2×/день) и рассылке digest. |

### 5.3. Безопасность

| ID | Требование |
|----|------------|
| NF-10 | Whitelist `telegram_id` для бота. |
| NF-11 | API-ключи только в `.env`, не в репозитории. |
| NF-12 | Маскирование токенов в URL при сохранении и отображении. |
| NF-13 | Санитизация контента страниц перед отправкой в LLM (prompt injection): (1) весь fetched-контент передаётся только внутри `user`-сообщения как чётко размеченный блок данных (например, `<page_content>...</page_content>`), никогда не смешивается с системными инструкциями; (2) system-prompt явно указывает игнорировать любые инструкции, найденные внутри `<page_content>`; (3) выходной JSON валидируется по схеме (`description`, `tags`, `confidence`) — любые лишние поля/текст вне схемы отбрасываются; (4) `tags` дополнительно фильтруются через `tag_normalizer` (allowlist символов, макс. длина), результат никогда не исполняется и не используется в SQL напрямую (только через ORM/параметризованные запросы). |
| NF-14 | Дашборд на MVP: Basic Auth через nginx (не открывать в публичный интернет без защиты). |

### 5.4. Масштабируемость

- Расчётная нагрузка: до 50 ссылок/день, до 10 Q&A/день, до 5 research-отчётов/день.
- Стек рассчитан на рост до ~5000 ссылок без смены архитектуры.

## 6. Архитектура

### 6.1. Компоненты

```
┌─────────────────────────────────────────────────┐
│  VPS (Ubuntu)                                   │
│                                                 │
│  nginx (SSL, reverse proxy, Basic Auth)         │
│    ├── /              → Dashboard (FastAPI)     │
│    └── /api           → REST API                │
│                                                 │
│  PostgreSQL + pgvector                            │
│  Redis                                          │
│  Celery worker + Celery Beat                    │
│  Telegram Bot (aiogram 3, webhook)              │
└─────────────────────────────────────────────────┘
         │                    │
         ▼                    ▼
    OpenAI API          Tavily / Serper API
```

### 6.2. Технологический стек

| Компонент | Технология |
|-----------|------------|
| Bot | Python 3.12, aiogram 3, webhook (через nginx, единый SSL-сертификат с дашбордом; long polling не используется на проде — лишний процесс и задержки при батч-нагрузке) |
| API + Dashboard | FastAPI, Jinja2, HTMX |
| Очередь | Celery + Redis |
| БД | PostgreSQL 16 + pgvector |
| LLM | OpenAI API |
| Поиск (research) | Tavily или Serper |
| Деплой | Docker Compose |
| SSL | Let's Encrypt (certbot) |

### 6.3. Поток обработки ссылки

```
Сообщение (чат / личка)
  → extract URLs
  → raw_messages (идемпотентно)
  → Worker:
      1. normalize URL → url_hash
      2. есть в links?
         ├─ да  → link_sources++, пересчёт счётчиков, ответ бота
         └─ нет → fetch meta
               → OpenAI (description, tags)
               → normalize tags
               → embedding
               → INSERT links + link_sources + link_tags
               → ответ бота
```

### 6.4. Маршрутизация сообщений в личке

```
Сообщение в личку
  ├─ содержит URL        → добавление ссылки
  ├─ команда (/ask и др.) → соответствующий handler
  ├─ текст без URL       → Q&A (RAG)
  └─ иное                → /help
```

## 7. Модель данных

### 7.1. Таблицы

```sql
-- Ссылки
CREATE TABLE links (
    id              SERIAL PRIMARY KEY,
    url             TEXT NOT NULL,
    normalized_url  TEXT NOT NULL,
    url_hash        VARCHAR(64) UNIQUE NOT NULL,
    title           TEXT,
    description     TEXT,
    domain          VARCHAR(255),
    favicon_url     TEXT,
    status          VARCHAR(20) DEFAULT 'pending',  -- pending|fetching|processing|done|failed
    fetch_error     TEXT,
    source_count    INT DEFAULT 1,
    unique_senders  INT DEFAULT 1,
    priority_score  FLOAT DEFAULT 0,
    is_hidden       BOOLEAN DEFAULT FALSE,
    embedding       vector(1536),
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    processed_at    TIMESTAMPTZ
);

-- Источники (контекст из Telegram)
CREATE TABLE link_sources (
    id              SERIAL PRIMARY KEY,
    link_id         INT REFERENCES links(id),
    chat_id         BIGINT,
    chat_title      TEXT,
    message_id      BIGINT,
    sender_id       BIGINT NOT NULL,
    sender_name     TEXT,
    message_text    TEXT,
    reply_to_text   TEXT,
    forwarded_from  TEXT,
    source_type     VARCHAR(10) NOT NULL,  -- group | direct
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Сырые сообщения (очередь)
CREATE TABLE raw_messages (
    id              SERIAL PRIMARY KEY,
    chat_id         BIGINT NOT NULL,
    message_id      BIGINT NOT NULL,
    sender_id       BIGINT,
    text            TEXT,
    entities_json   JSONB,
    source_type     VARCHAR(10) NOT NULL,
    processed       BOOLEAN DEFAULT FALSE,
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (chat_id, message_id)
);

-- Теги
CREATE TABLE tags (
    id              SERIAL PRIMARY KEY,
    name            VARCHAR(100) UNIQUE NOT NULL,
    slug            VARCHAR(100) UNIQUE NOT NULL
);

CREATE TABLE link_tags (
    link_id         INT REFERENCES links(id),
    tag_id          INT REFERENCES tags(id),
    PRIMARY KEY (link_id, tag_id)
);

-- Синонимы тегов (нормализация без хардкода в коде)
CREATE TABLE tag_synonyms (
    id              SERIAL PRIMARY KEY,
    raw_value       VARCHAR(100) UNIQUE NOT NULL,  -- напр. 'ии', 'нейросети'
    canonical_tag   VARCHAR(100) NOT NULL           -- напр. 'ai'
);

-- LLM-отчёты «Собрать ещё»
CREATE TABLE research_reports (
    id              SERIAL PRIMARY KEY,
    link_id         INT REFERENCES links(id),
    topic           TEXT,
    report_md       TEXT NOT NULL,
    sources_json    JSONB,
    model           VARCHAR(50),
    tokens_used     INT,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Тематические подборки
CREATE TABLE collections (
    id              SERIAL PRIMARY KEY,
    title           TEXT NOT NULL,
    theme           VARCHAR(100),
    period_start    DATE,
    period_end      DATE,
    summary_md      TEXT NOT NULL,
    link_ids        JSONB,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Логи Q&A
CREATE TABLE qa_logs (
    id              SERIAL PRIMARY KEY,
    user_id         BIGINT,
    question        TEXT NOT NULL,
    answer_md       TEXT NOT NULL,
    matched_link_ids JSONB,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
```

### 7.2. Индексы

```sql
CREATE INDEX idx_links_url_hash ON links(url_hash);
CREATE INDEX idx_links_priority ON links(priority_score DESC);
CREATE INDEX idx_links_created ON links(created_at DESC);
CREATE INDEX idx_links_status ON links(status);
CREATE INDEX idx_link_sources_link ON link_sources(link_id);
-- ivfflat эффективен от десятков тысяч векторов; при ожидаемом объёме MVP (до ~5000 ссылок)
-- brute-force cosine (без индекса) будет не медленнее и не потребует настройки lists/probes.
-- Индекс добавить, когда база вырастет за пределы ~10-20k строк.
CREATE INDEX idx_links_embedding ON links USING ivfflat (embedding vector_cosine_ops);
```

## 9. Промпты OpenAI (черновики)

### 9.1. Описание и теги


## 12. План разработки

| Неделя | Задачи | Результат |
|--------|--------|-----------|
| **1** | BotFather, aiogram, extract URLs, PostgreSQL, модели, whitelist, raw_messages | Бот принимает сообщения, ссылки в очереди |

| **6** | Docker Compose, деплой на VPS, nginx, SSL, мониторинг, полировка | Продакшен |

## 14. Риски и митигация

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Бот не видит сообщения в группе | Средняя | Privacy Mode off, проверка прав |
| Плохие описания (paywall, 404) | Высокая | Fallback на контекст чата, статус `fetch_failed` |
| Хаос в тегах | Средняя | Нормализация, автокомплит, ручное слияние |
| Команда не пользуется дашбордом | Средняя | Digest в TG, уведомления бота, топ на главной |
| Рост стоимости OpenAI | Низкая | mini для тегов, кэш отчётов, лимиты |
| Дашборд без auth в открытом доступе | Высокая | Basic Auth на nginx до интеграции в сайт |

## 15. Вне скоупа MVP

- Авторизация через сайт (Telegram Login, JWT).
- Userbot и чтение истории до добавления бота.
- Экспорт в Notion / Obsidian.
- Мобильное приложение.

### 15.1. Кандидаты в v1.1 (вероятно понадобятся раньше, чем через полгода)

- **Модерация шума через LLM-фильтр `is_useful`.** Поскольку F-01 сохраняет в очередь только сообщения с URL, риск засорения `raw_messages` ниже, чем при полном логировании чата — но если в ссылках начнёт попадаться нерелевантный шум (мемы, случайные репосты), фильтр стоит добавить одним из первых после MVP.
- **Дедупликация по смыслу (embedding similarity merge).** На старте разных URL, ведущих на один и тот же контент (мобильная/десктоп версия, реферальные обёртки, зеркала), может набраться заметная доля — стоит закладывать бюджет на эту задачу сразу после стабилизации точной (`url_hash`) дедупликации.
- **Real-time алерт о всплеске популярности.** Помимо еженедельного digest — уведомление в Telegram, когда `priority_score` ссылки пересекает порог (например, `unique_senders >= 3` в течение суток), чтобы не ждать понедельника ради действительно горячих материалов.

## 16. Критерии приёмки MVP

- [ ] Бот добавлен в чаты, собирает ссылки из групп и лички.
- [ ] Дубли не создают повторных записей, счётчики обновляются.
- [ ] Бот сообщает количество добавлений при каждой операции.
- [ ] У каждой ссылки есть описание и теги от OpenAI.
- [ ] Дашборд отображает ссылки с фильтрами, поиском и сортировкой по приоритету.
- [ ] Q&A отвечает на вопросы по базе с цитатами.
- [ ] «Собрать ещё» генерирует LLM-отчёт.
- [ ] Тематическая подборка генерируется еженедельно.
- [ ] Система развёрнута на VPS через Docker Compose.
- [ ] Whitelist и Basic Auth работают.

## 17. Глоссарий

| Термин | Определение |
|--------|-------------|
| **source** | Одно добавление ссылки (из чата или лички) |
| **source_count** | Сколько раз ссылку кинули суммарно |
| **unique_senders** | Сколько разных людей кинули ссылку |
| **priority_score** | Числовой вес популярности ссылки |
| **collection** | Тематическая подборка за период |
| **research report** | LLM-отчёт с внешними источниками по теме ссылки |
| **RAG** | Retrieval-Augmented Generation — поиск по базе + генерация ответа |
